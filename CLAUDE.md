# Проект: парсер коммерческой недвижимости Беларуси

Инструмент для лидогенерации: собирает объявления коммерческой недвижимости из
нескольких источников в один Excel с телефонами/адресами для обзвона.

Окружение: venv в `/Users/denis/realty_env`. Запуск Python: `./bin/python`.
Зависимости: `playwright`, `openpyxl`, `playwright-stealth`, `beautifulsoup4`+`lxml` (HTML-парсинг —
ПРЕДПОЧТИТЕЛЬНЕЕ регэкспов, в т.ч. в брифах ботам), `pypdf`. Системно: `poppler` (чтение PDF),
`tesseract`+`rus.traineddata` (OCR PDF-сканов — опц., только где надо: `auctions_common.pdf_text()`,
движок глобальный `~/.claude/ocr_pdf.py`), `git` (репозиторий инициализирован 01.06, первый коммит
сделан; venv/xlsx/photos в .gitignore).
Платформа: macOS.
**Бэкап проекта: Яндекс.Диск** — `~/Yandex.Disk.localized/realty_backup/` (в Finder показывается
как «Яндекс.Диск»). Синкается ВСЁ: код, все xlsx (включая `saved_realty.xlsx`), папка `photos/`,
CLAUDE.md, `.command`-лаунчер. iCloud-бэкап убран — пользователь предпочёл одну облачную копию
в Яндексе. Константа в коде: `BACKUP_DIR` в `collect_realty.py`.

## Фича «Сохранить» (избранные объекты)

Колонка «Сохранить» в основном файле (первая, всегда видна — закреплена вместе с «Тип»).
Пользователь ставит метку (`x`, `+`, `1`) против интересных объектов. Запуск
`./bin/python save_marked.py` или двойной клик по `Сохранить.command`:
- качает фото из колонки «Фото URL» в `photos/{хэш}/` (макс. 3 шт на объект);
- через бесплатный **OSM Overpass** (без API-ключей!) считает:
  - «Активность локации» (POI рядом в 300м: магазины/кафе/услуги/офисы);
  - «До транспорта, м» (ближайшая остановка/метро/станция);
- дописывает в отдельный `saved_realty.xlsx` (лист «Сохранённые») с путями к фото и гео.
- Уже сохранённые (по хэшу) пропускаются — можно запускать многократно.

**Важно для Overpass**: User-Agent должен быть НЕ браузерный (overpass-api.de отдаёт 406
на `Mozilla/...`). Используется `OVERPASS_UA = "realty-tool/1.0"`. Уже зашито.

**Координаты и Фото URL** извлекаются при сборе:
- **kufar** — из `__NEXT_DATA__` (`coordinates`, `images[].path` → `https://rms.kufar.by/v1/gallery/{path}`). Фото ~98%, коорд 100%.
- **megapolis** — из `data-coord` и `data-src` в rItem (lazy-load, **именно `data-src`**, не `src` — `src` это `blank_*.png`-заглушка). Относительные пути дополняются `https://megapolis-real.by/`. Фото 100%, коорд 99% у пересобранных.
- **realt** — фото НЕ извлекаются (см. ниже). Координаты — да, через `page.evaluate` ищу `lat,lng` в innerText/data-атрибутах (~65%). Когда коорд нет, save_marked геокодит адрес через OSM Nominatim.

**Realt-фото — известный TODO**: эксперимент показал, что единственный `a.realt.by/openximg/...jpg`,
видимый в `<img>` сразу после загрузки, — это общая og:image-заглушка (тот же URL на ВСЕХ карточках).
Реальные фото галереи лежат глубже (CSS background-image / lazy через JS-компонент / отдельный API).
Чтобы реально вытащить, нужно либо более долгий wait + точечный селектор галереи,
либо реверс XHR-запросов. **Пока realt-объекты сохраняются без фото.**

Старые `realty_parser_v3..v7.py`, `patch_v8.py` — легаси, перемещены в `legacy/` (31.05), не используются.

## SwiftUI-приложение (`RealtyApp/`)

Нативный GUI-фронтенд для Python-парсера. Папка `~/realty_env/RealtyApp/` —
SPM-проект, открывается через `xed Package.swift`. min macOS 13.
Зависимость: `CoreXLSX` (только чтение). Запись xlsx из Swift не делается —
для «сохранить выбранное» вызывается `save_marked.py --hashes h1,h2,...`,
для удаления из сохранённых — генерируется на лету тонкий Python-помощник.
**Sandbox должен быть отключён** в Xcode (иначе нет доступа ни к Process,
ни к `~/realty_env`). Архитектура MVVM: `AppState` (@MainActor ObservableObject),
`XLSXService` (CoreXLSX), `ProcessRunner` (Process+Pipe со стримингом),
вьюхи `ContentView` / `MainTableView` / `SavedView` / `LogPanel`.

Изменение в `save_marked.py`: добавлен argparse и флаг `--hashes h1,h2,...` —
позволяет SwiftUI-приложению передать выбранные объекты без записи метки в xlsx.

## Веб-дашборд «Консоль обзвона» (`web/`, 11.06.2026)

Веб-фронтенд для обзвона поверх `commercial_realty.xlsx`. Сделан скилом **impeccable**
(`~/.agents/skills/impeccable`, установлен юзером 08.06; product-регистр, OKLCH, системный
шрифт, все состояния, тёмная тема). Запуск: двойной клик `Дашборд.command` (в корне) →
`export_data.py` (xlsx→`web/data.js`) → `web/server.py` (локальный http+API) → браузер.
**Открывать только через сервер (http://localhost:8765), не `file://`** — иначе Safari
блокирует ссылки/кнопки (нужен бэкенд + http-origin для буфера/popup).

- **`export_data.py`** — листы «Продажа»/«Аренда» (шапка=стр.2, данные с 3) → `data.js`
  (`window.LISTINGS`/`META`). Чистит город (`г. Г. Минск`→`Минск`), `0 $`→цена скрыта,
  кладёт `sheet`+`row` (номер строки Excel) для перехода. `data.js` 7+МБ — рендер порциями по 48.
- **`server.py`** — статика + API (только 127.0.0.1, индекс по хэшу из data.js):
  `POST /api/save` (офлайн-копия объявления; фото параллельно, 6 потоков — иначе Safari
  обрывал fetch на минутных сохранениях), `GET /api/saved` (хэши сохранённых — карточки
  сразу зелёные после перезагрузки), `POST /api/reveal` (Excel: активирует книгу/лист,
  выделяет ВСЮ строку A:AE и скроллит к ней; returncode osascript ПРОВЕРЯЕТСЯ — при отказе
  Automation открывает файл + подсказка про Настройки→Автоматизация), `GET /api/ping`
  (+CORS — страница с file:// сама находит сервер и redirect'ится), `POST /api/update` +
  `GET /api/update/status` + `POST /api/update/stop` — **обновление через `_stream()`**, тело POST
  `{target}` (валидируется whitelist'ом). **ОДНА кнопка «Обновить» в шапке (01.07.2026) → target
  `all`** = полный прогон через `_update_steps()`: `collect_realty.py` (realt/megapolis/kufar/
  **gohome/byrealty**) → `kufar_phones.py --limit KUFAR_PHONE_LIMIT` → **`collect_geo.py`
  (domovita+edc, БЕЗ VPN)** → `collect_auctions.py` (10 площадок) → `collect_banks.py` +
  `belretail_phones.py` → один ре-экспорт. Гео-шаг и kufar/belretail-телефоны сами пропустятся,
  если IP не белорусский (под VPN соберут 0, остальное не ломают). Раздельные targets
  (`realty`/`geo`/`auctions`/`banks`) whitelist оставлены (CLI/будущее), но в UI кнопка одна.
  Один общий JOB/status/stop — два сбора разом нельзя (оба пишут в xlsx). Всё со стримом лога.
  Каждый шаг — отдельный процесс → terminate чистый. После ре-экспорта — шаг прогрева
  фото-кэша (`prefetch_photos.py --limit 2000`, идемпотентно: новые объекты докачиваются).
  **Фото карточек — через локальный кэш `/img?hash=` (22.06.2026), НЕ напрямую с CDN.**
  Корень старой боли «серые карточки / обрезанные фото»: браузерная ПАЧКА из ~48 параллельных
  запросов к `rms.kufar.by` под VPN (иностранный IP) СТОПОРИТСЯ (kufar — 43% базы) → фото
  не догружались (серое) или грузились наполовину (визуально «обрезано»). Диагноз проверен:
  последовательная СЕРВЕРНАЯ загрузка kufar-gallery под тем же VPN = 6/6 ОК (~0.9с/фото);
  стопорит именно браузерный параллельный всплеск. Решение: фронт грузит `<img src="/img?hash=">`
  (см. `photoTag` в app.js), сервер отдаёт фото с диска `web/photos_cache/{хэш}.{jpg,png,webp}`
  (gitignore), при промахе докачивает ПОЛНОразмер (kufar `list_thumbs_2x`→`gallery` 873×690;
  realt/прочее без фото → og:image с деталки) ПОСЛЕДОВАТЕЛЬНО через `FETCH_SEM=3` — пачка
  источник не злит. `.none`-файл = негативный кэш (фото нет, повторно не дёргаем).
  Браузер берёт фото с localhost → мгновенно, целиком, полный размер, НЕ зависит от IP/VPN.
  **Прогрев:** `web/prefetch_photos.py` (или `Прогрев фото.command`) — качает фото ВСЕХ
  объектов в кэш (идемпотентно, resumable, ~3/с под VPN, полный прогон ~1.5ч). Запускать
  под VPN можно (последовательно kufar не банит) или без VPN (быстрее). og:image отдают и
  крупные аукционы: ipmtorgi/cpo ✅ (konfiskat/torgi24/gki/mgcn ❌ — у них og нет).
  **megapolis-фото — кэш-URL протухает (22.06):** сохранённый при сборе `photo`
  (`assets/cache/images/...-600x400.jpeg`) megapolis периодически чистит → 404 → карточка
  «фото недоступно», хотя на самом объявлении фото ЕСТЬ. Фикс: `_photo_urls()` (бывш.
  `_best_photo_url`) отдаёт СПИСОК кандидатов — сперва `photo`, при промахе **og:image деталки**
  (живой оригинал `assets/images/userfiles/...`); `fetch_photo` идёт по списку до первого
  успеха, `.none` ставит лишь когда ВСЕ кандидаты пусты. ⚠ megapolis БАНИТ всплеск (TCP-дроп,
  BrokenPipe) → массовый прогрев залпом наставит ЛОЖНЫХ `.none`. Греть megapolis ТОЛЬКО медленно:
  `prefetch_photos.py --sources megapolis-real.by --workers 1 --delay 4` (новый флаг `--delay`)
  или **`Прогрев megapolis.command`** — лучше БЕЗ VPN (свой IP добрее). Ложные `.none` от бана
  снимаются удалением `{хэш}.none` у megapolis-хэшей и повторным аккуратным прогревом.
  **realt-фото в карточках были «лентой» (22.06, фикс):** `.lead__media img{height:100%}` НЕ
  резолвится от родителя с `aspect-ratio` (без явной высоты браузер трактует как `auto`) → фото
  падало в натуральную высоту, серые поля. Фикс в styles.css: `img{position:absolute;inset:0}`

  Старый `GET /api/photo?hash=` (realt og:image, `photo_cache.json`) оставлен, но карточки
  теперь идут через `/img`; полный размер для СОХРАНЕНИЯ по-прежнему качает `save`.
  Телефон в сохранёнке: из строки таблицы, иначе regex по тексту объявления

## Главная команда

```
./bin/python collect_realty.py                    # все источники, инкрементально
./bin/python collect_realty.py --sources megapolis,kufar
./bin/python collect_realty.py --city belarus     # kufar по всей РБ (дефолт minsk)
./bin/python collect_realty.py --max-pages 5      # ограничить глубину
./bin/python collect_realty.py --full             # полный перепрогон (осторожно: затирает базу до собранного)
./bin/python doctor.py                            # health-check: кто из источников жив + доступны ли телефоны kufar (по IP)
```
Standalone-парсеры (`megapolis_parser.py`, `kufar_parser.py`, `realty_parser_v8.py --debug`)
работают по отдельности, пишут в свои файлы.
